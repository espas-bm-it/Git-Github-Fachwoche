Einen Branch erstellen

Wenn Sie an einem Projekt arbeiten, werden Sie zu einem bestimmten Zeitpunkt eine Reihe verschiedener Funktionen oder Ideen in Arbeit haben - von denen einige bereits fertig sind und andere noch nicht. Verzweigungen helfen Ihnen, diesen Arbeitsablauf zu verwalten. Wenn Sie einen Zweig in Ihrem Projekt erstellen, schaffen Sie eine Umgebung, in der Sie neue Ideen ausprobieren können. Änderungen, die Sie in einer Verzweigung vornehmen, wirken sich nicht auf die main Branch aus. Sie können also experimentieren und Änderungen einbringen, in der Gewissheit, dass Ihre Verzweigung erst dann zusammengeführt wird, wenn sie von jemandem, mit dem Sie zusammenarbeiten, geprüft werden kann.

ProTipp
Verzweigungen sind ein zentrales Konzept in Git, und der gesamte GitHub-Ablauf basiert darauf. Es gibt nur eine Regel: Alles im main Branch ist immer einsatzfähig. Aus diesem Grund ist es extrem wichtig, dass Ihr neuer Zweig aus dem Hauptzweig erstellt wird, wenn Sie an einem Feature oder einer Korrektur arbeiten. Ihr Zweigname sollte aussagekräftig sein (z.B. refactor-authentication, user-content-cache-key, make-retina-avatars), damit andere sehen können, woran gearbeitet wird.

Commits hinzufügen

Sobald Ihr Zweig erstellt wurde, ist es an der Zeit, Änderungen vorzunehmen. Jedes Mal, wenn Sie eine Datei hinzufügen, bearbeiten oder löschen, machen Sie einen Commit und fügen sie zu Ihrem Zweig hinzu. Durch das Hinzufügen von Commits können Sie Ihren Fortschritt bei der Arbeit an einem Funktionszweig verfolgen. Commits erstellen auch eine transparente Historie Ihrer Arbeit, die andere verfolgen können, um zu verstehen, was Sie getan haben und warum. Jeder Commit hat eine zugehörige Commit-Nachricht, eine Beschreibung, die erklärt, warum eine bestimmte Änderung vorgenommen wurde. Darüber hinaus wird jeder Commit als eigene Änderungseinheit betrachtet. So können Sie Änderungen rückgängig machen, wenn ein Fehler gefunden wird oder wenn Sie eine andere Richtung einschlagen wollen.

ProTipp
Commit-Nachrichten sind wichtig, vor allem weil Git Ihre Änderungen verfolgt und sie als Commits anzeigt, sobald sie auf den Server übertragen wurden. Indem Sie klare Commit-Nachrichten schreiben, können Sie es anderen Personen erleichtern, mitzuverfolgen und Feedback zu geben.

Eine Pull Request öffnen

Pull Requests initiieren eine Diskussion über Ihre Commits. Da sie eng mit dem zugrunde liegenden Git-Repository verbunden sind, kann jeder genau sehen, welche Änderungen zusammengeführt werden, wenn er Ihre Anfrage akzeptiert. Sie können zu jedem Zeitpunkt des Entwicklungsprozesses einen Pull Request eröffnen: wenn Sie wenig oder gar keinen Code haben, aber einige Screenshots oder allgemeine Ideen mit anderen teilen möchten, wenn Sie nicht weiterkommen und Hilfe oder Rat brauchen oder wenn Sie bereit sind, dass jemand Ihre Arbeit überprüft. Indem Sie das @mention-System von GitHub in Ihrer Pull Request-Nachricht verwenden, können Sie bestimmte Personen oder Teams um Feedback bitten, egal ob sie sich am Ende des Flurs oder zehn Zeitzonen entfernt befinden.

ProTipp
Pull Requests sind nützlich, um zu Open-Source-Projekten beizutragen und um Änderungen an gemeinsam genutzten Repositories zu verwalten. Wenn Sie ein Fork & Pull Modell verwenden, bieten Pull Requests eine Möglichkeit, die Projektbetreuer über die Änderungen zu informieren, die sie berücksichtigen sollen. Wenn Sie ein Shared Repository Modell verwenden, helfen Pull Requests bei der Überprüfung des Codes und der Diskussion über vorgeschlagene Änderungen, bevor diese in den Hauptzweig zusammengeführt werden.

Diskutieren und überprüfen Sie Ihren Code

Sobald ein Pull Request geöffnet wurde, kann die Person oder das Team, das Ihre Änderungen überprüft, Fragen oder Kommentare haben. Vielleicht entspricht der Codierungsstil nicht den Projektrichtlinien, der Änderung fehlen Unit-Tests, oder vielleicht sieht alles gut aus und die Requisiten sind in Ordnung. Pull Requests sind dazu gedacht, diese Art von Konversation zu fördern und festzuhalten. Sie können auch im Lichte der Diskussion und des Feedbacks zu Ihren Commits weiter an Ihrem Zweig arbeiten. Wenn jemand bemerkt, dass Sie etwas vergessen haben zu tun, oder wenn es einen Fehler im Code gibt, können Sie ihn in Ihrem Zweig beheben und die Änderung hochschieben. GitHub zeigt Ihre neuen Commits und alle zusätzlichen Rückmeldungen, die Sie erhalten haben, in der vereinheitlichten Pull Request-Ansicht an.

ProTipp
Pull Request-Kommentare werden in Markdown geschrieben, so dass Sie Bilder und Emoji einbetten, vorformatierte Textblöcke und andere einfache Formatierungen verwenden können.

Bereitstellen (Deploy)

Mit GitHub können Sie von einem Zweig aus endgültige Tests in der Produktion durchführen, bevor Sie ihn mit dem Hauptzweig zusammenführen. Sobald Ihre Pull-Anfrage geprüft wurde und der Zweig Ihre Tests bestanden hat, können Sie Ihre Änderungen bereitstellen, um sie in der Produktion zu überprüfen. Wenn Ihr Zweig Probleme verursacht, können Sie ihn zurücksetzen, indem Sie den bestehenden Hauptzweig in der Produktion einsetzen. Verschiedene Teams können unterschiedliche Bereitstellungsstrategien haben. Für einige ist es vielleicht am besten, in einer speziell eingerichteten Testumgebung bereitzustellen. Für andere kann die direkte Bereitstellung in der Produktion die bessere Wahl sein, wenn man die anderen Elemente ihres Arbeitsablaufs berücksichtigt.

Zusammenführen (Merge)

Nachdem Ihre Änderungen in der Produktion überprüft wurden, ist es nun an der Zeit, Ihren Code in den Hauptzweig einzubinden. Nach dem Zusammenführen bewahren Pull Requests eine Aufzeichnung der historischen Änderungen an Ihrem Code. Da sie durchsuchbar sind, kann jeder in der Zeit zurückgehen, um zu verstehen, warum und wie eine Entscheidung getroffen wurde.

ProTip
Indem Sie bestimmte Schlüsselwörter in den Text Ihres Pull Requests einfügen, können Sie Probleme mit dem Code in Verbindung bringen. Wenn Ihr Pull Request zusammengeführt wird, werden auch die damit verbundenen Probleme geschlossen. Wenn Sie z.B. den Satz Closes #32 eingeben, wird Issue Nummer 32 im Repository geschlossen.
